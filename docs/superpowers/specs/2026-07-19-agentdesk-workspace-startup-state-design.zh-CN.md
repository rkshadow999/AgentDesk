# AgentDesk 无工作区启动状态修复设计

## 问题

AgentDesk 在未选择工作区时，Web 前端仍会立即请求 `session/list`。宿主因为缺少工作区拒绝该请求，却把原因统一映射为“引擎操作失败”。第一次选择工作区时，前端又不会重新请求会话列表，因此错误卡会继续残留。

这会把正常的初始状态错误地表现为引擎损坏。

## 方案比较

1. 只改前端：无工作区时不请求会话列表。界面会正确，但旧前端或竞态请求仍可能让宿主返回伪引擎错误。
2. 只改宿主：无工作区时返回空列表。不会报错，但前端仍会显示无意义的加载状态，也缺少明确的工作区入口。
3. 前后端双保险：前端门控请求并提供“选择工作区”空状态；宿主对无工作区的列表请求返回空列表。该方案覆盖正常流程、竞态和版本不一致，采用此方案。

## 行为

- 未选择工作区时不发送 `session/list`，不启动 sidecar，不显示错误或重试卡。
- 会话区域显示“选择工作区”操作，点击后发送 `workspace/select`。
- 第一次收到 `workspace/selected` 后立即请求会话列表，并清除先前列表错误。
- 从一个工作区切换到另一个工作区时，继续清空当前会话上下文并重新请求列表。
- 宿主收到无工作区的 `session/list` 时返回与请求 ID 关联的空 `session/list/changed`，不发布 `session/list/error`，也不创建 sidecar。
- 本次不自动恢复或信任上次工作区；该行为涉及单独的安全和产品决策。

## 测试

- Web：无初始工作区时不发送列表请求，显示选择按钮；按钮发送 `workspace/select`。
- Web：首次工作区选择会发送列表请求，并从无工作区空状态进入加载状态。
- Web：已有工作区的搜索、归档和错误重试行为保持不变。
- C#：无工作区列表请求返回空关联结果，工厂不启动 sidecar。
- 打包后：冷启动截图不得出现“引擎操作失败”；选择工作区后显示“引擎已连接”。
